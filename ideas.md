## فیچرهای جدید

### مدیریت لایسنس

6. **لغو خودکار لایسنس پس از استرداد**: وقتی وضعیت سفارش به `refunded` تغییر می‌کنه، لایسنس رو به‌صورت خودکار غیرفعال کن (صفر کردن device ها).
   > **تأثیر بر سرعت:** بدون تأثیر روی کاربران عادی یا فرانت‌اند. فقط لحظه‌ای که ادمین سفارش رو مسترد می‌کنه یه API call اضافه می‌شه که چند صدم ثانیه طول می‌کشه. برای checkout و صفحات عمومی سایت کاملاً بی‌تأثیره.

7. **تمدید لایسنس**: دکمه‌ای در صفحه سفارش ادمین که تاریخ انقضای لایسنس رو از طریق API تمدید کنه.
   > **تأثیر بر سرعت:** خنثی در همه حالت‌ها. یه API call on-demand هنگام کلیک دکمه در ادمین. کاربران فرانت‌اند اصلاً درگیر نمی‌شند.

8. **نمایش تعداد دستگاه‌های فعال**: در باکس سفارش ادمین تعداد دستگاه‌هایی که لایسنس را فعال کرده‌اند نشان بده.
   > **تأثیر بر سرعت:** تأثیر منفی قابل‌توجه روی صفحه ویرایش سفارش در ادمین (اگه بدون caching پیاده بشه). برای هر بار باز شدن صفحه سفارش، یه API call به اسپات پلیر زده می‌شه که ۳۰۰–۸۰۰ms تأخیر ایجاد می‌کنه. راه‌حل: باید با AJAX lazy load یا transient با TTL کوتاه (~۵ دقیقه) پیاده بشه. برای فرانت‌اند و کاربران عادی بی‌تأثیره.

9. **Webhook از اسپات پلیر**: یه endpoint در وردپرس بساز که رویدادهای اسپات پلیر (مثل فعال‌سازی دستگاه) رو دریافت و در متادیتای سفارش ذخیره کنه.
   > **تأثیر بر سرعت:** خنثی برای کاربران سایت. فقط وقتی اسپات پلیر webhook می‌فرسته یه WordPress bootstrap اجرا می‌شه که روی سرور باره، نه روی تجربه کاربر. یه DB write ساده هم انجام می‌شه.

### تجربه کاربر

10. **ارسال کلید لایسنس در ایمیل**: پس از ساخت لایسنس، کلید رو در ایمیل تأییدیه سفارش ووکامرس یا یه ایمیل جداگانه بفرست.
    > **تأثیر بر سرعت:** اگه با WP Cron یا در پس‌زمینه انجام بشه: خنثی. اگه sync باشه: می‌تونه چند صدم ثانیه به زمان پردازش سفارش اضافه کنه، ولی کاربر این رو احساس نمی‌کنه چون صفحه تأیید سفارش قبلاً لود شده. برای فرانت‌اند عادی بی‌تأثیره.

11. **نمایش تاریخ انقضا به کاربر**: اگه API تاریخ انقضا برمی‌گردونه، در صفحه دوره‌های کاربر نمایش بده.
    > **تأثیر بر سرعت:** کاملاً بستگی به پیاده‌سازی داره. اگه تاریخ انقضا را در متادیتا ذخیره کنیم (هنگام ساخت لایسنس): خنثی، فقط یه DB read ساده‌ست. اگه برای نمایش API call بزنیم: تأثیر منفی جدی روی صفحه دوره‌های کاربر برای همه کاربران (۳۰۰–۸۰۰ms به ازای هر لایسنس). پیاده‌سازی صحیح: ذخیره local.

12. **غیرفعال‌سازی دستگاه توسط کاربر**: اگه API امکانش رو داشته باشه، کاربر بتونه از حساب کاربری‌اش یه دستگاه رو غیرفعال کنه.
    > **تأثیر بر سرعت:** خنثی روی صفحات عادی. فقط لحظه کلیک کاربر یه API call اجرا می‌شه. هیچ باری روی فرانت‌اند منفعلانه نداره.

    > **تأثیر بر سرعت:** تأثیر منفی بسیار جزئی هنگام ساخت/خطای لایسنس (یه INSERT). صفحه مشاهده لاگ در ادمین یه SELECT ساده‌ست. برای فرانت‌اند و کاربران عادی کاملاً بی‌تأثیره. با گذشت زمان اگه لاگ‌ها تمیز نشند جدول بزرگ می‌شه، پس نیاز به retention policy داره.

### یکپارچه‌سازی

23. **WooCommerce Subscriptions**: وقتی اشتراک تجدید می‌شه، لایسنس رو تمدید کن؛ وقتی لغو می‌شه، غیرفعالش کن.
    > **تأثیر بر سرعت:** بدون تأثیر روی کاربران. هر دو عملیات (تمدید و لغو) توسط WP Cron در پس‌زمینه اجرا می‌شند. یه API call اضافه در جریان پردازش renewal. فرانت‌اند اصلاً درگیر نمی‌شه.

24. **WooCommerce Memberships**: دسترسی به لایسنس رو به عضویت گره بزن.
    > **تأثیر بر سرعت:** تأثیر منفی جزئی روی فرانت‌اند. هر بار که کاربر صفحه دوره‌ها رو باز می‌کنه یه چک عضویت اضافه می‌شه. اگه Memberships API کند باشه این تأثیر محسوس‌تره. می‌تونه ۲۰–۱۰۰ms اضافه کنه به صفحه دوره‌های کاربر.

25. **WP-CLI**: کامندهایی مثل `wp spotplayer create-license <order_id>` و `wp spotplayer disable-license <license_id>` برای سناریوهای اتوماسیون.
    > **تأثیر بر سرعت:** صفر. CLI فقط از خط فرمان اجرا می‌شه و هیچ overhead ای روی وب‌سایت نداره. در واقع می‌تونه به عنوان ابزار مدیریت offline بدون بار روی وب سرور استفاده بشه.

26. **REST API**: یه endpoint مثل `GET /wp-json/spotplayer/v1/license/<order_id>` برای ادغام با سیستم‌های خارجی.
    > **تأثیر بر سرعت:** تأثیر منفی بسیار جزئی. ثبت route جدید چند میکروثانیه به bootstrap وردپرس اضافه می‌کنه. هنگام فراخوانی endpoint یه WP bootstrap کامل + DB query اجرا می‌شه. برای فرانت‌اند و صفحات عادی کاملاً بی‌تأثیره.

## بهبود و بهینه‌سازی

### کد و معماری

27. **جایگزینی `eval()`**: تابع `spot_woo_license_data_eval` کد PHP رو با `eval()` اجرا می‌کنه. می‌شه از یه closure یا parser محدود استفاده کرد، یا حداقل خروجی رو بیشتر validate کرد.
    > **تأثیر بر سرعت:** تأثیر مثبت جزئی در همه جایی که لایسنس ساخته می‌شه. `eval()` نمی‌تونه توسط PHP OPcache کمپایل و کش بشه، یعنی هر بار مجدداً parse می‌شه. جایگزینی با closure کد رو OPcache-friendly می‌کنه.

28. **کشینگ با Transient**: نتیجه `spot_request_license_get` رو برای مدت کوتاهی (مثلاً ۵ دقیقه) در transient ذخیره کن تا تعداد درخواست‌های API کم بشه.
    > **تأثیر بر سرعت:** تأثیر مثبت قابل‌توجه روی پنل ادمین. هر بار که صفحه سفارش باز می‌شه به جای یه API call خارجی (۳۰۰–۸۰۰ms) از cache می‌خونه (زیر ۵ms).

29. **Retry خودکار**: اگه ساخت لایسنس با خطای شبکه شکست خورد (نه خطای منطقی)، یه بار دیگه با تأخیر کوتاه امتحان کن.
    > **تأثیر بر سرعت:** در حالت عادی (بدون خطا): خنثی. در صورت خطای شبکه: تأخیر ۱–۳ ثانیه به پردازش سفارش اضافه می‌کنه.

### 🔁 بازیابی خودکار لایسنس‌های ناموفق

37. **Cron بازیابی لایسنس‌های جا افتاده**: یه WP Cron هر ۱ ساعت اجرا بشه و سفارشاتی که بیش از ۱۵ دقیقه از تکمیلشون گذشته ولی هنوز `_spotplayer_data` ندارند رو پیدا کنه و مجدداً تلاش کنه لایسنس بسازه. این مستقیم‌ترین راه‌حل برای سفارشاتیه که به خاطر قطعی اینترنت یا خطای لحظه‌ای بی‌لایسنس موندند.
    > **کجا تأثیر داره:** پس‌زمینه سرور، هیچ تأثیری روی UX نداره. درصد موفقیت لایسنس‌ها را به نزدیک ۱۰۰٪ می‌رسونه.

### 📲 ارتباط با کاربر

38. **اطلاع‌رسانی SMS پس از ساخت لایسنس**: پس از ایجاد موفق لایسنس، یه پیامک حاوی کلید لایسنس برای کاربر ارسال کن. از افزونه‌های SMS متداول ایران (Digits SMS، Melipayamak، Kavenegar hook) پشتیبانی کن. این مهم‌ترین کانال ارتباطی در بازار ایرانه.
    > **کجا تأثیر داره:** فقط لحظه ساخت لایسنس، async. هیچ تأثیری روی UX سایت نداره.

39. **اعلان ادمین هنگام خطای لایسنس**: الان خطاها فقط توی notice ادمین نشون داده می‌شند که اگه ادمین اون لحظه در پنل نباشه می‌بینه. باید یه ایمیل هم به ادمین سایت فرستاده بشه که «لایسنس سفارش #X ساخته نشد» تا هیچ خریدی بی‌پاسخ نمونه.
    > **کجا تأثیر داره:** فقط در موارد خطا، async. ریسک از دست دادن خرید مشتری رو کاهش می‌ده.

### 🎁 قابلیت هدیه دادن دوره

40. **Gift License — خرید به نام دیگری**: در صفحه محصول یا checkout، گزینه‌ای برای «خرید برای دیگری» اضافه کن که کاربر بتونه نام و شماره گیرنده رو وارد کنه. لایسنس با مشخصات گیرنده (نه خریدار) ساخته می‌شه. این یه differentiator بزرگه که هیچ افزونه اسپات پلیری فعلاً نداره.
    > **کجا تأثیر داره:** checkout (یه فیلد اضافی) + منطق ساخت لایسنس. تأثیر سرعتی خنثی.

### 🔗 بهبود باکس ادمین سفارش

41. **لینک مستقیم به پنل اسپات پلیر**: در باکس سفارش ادمین، کنار شناسه لایسنس یه دکمه «مشاهده در پنل» بذار که به `https://panel.spotplayer.ir/license/edit/{id}` لینک بده. الان شناسه هست ولی باید copy کنی و بری پنل.

42. **Bulk Action در لیست سفارشات**: «ساخت لایسنس اسپات پلیر» رو به عنوان یه Bulk Action به لیست سفارشات ووکامرس اضافه کن. ادمین می‌تونه چند سفارش رو tick کنه و یه‌جا لایسنسشون رو بسازه بدون اینکه وارد هر سفارش بشه.
    > **کجا تأثیر داره:** فقط در لیست سفارشات ادمین هنگام استفاده.

43. **disable/enable لایسنس مستقیم از سفارش**: دکمه‌ای در باکس سفارش ادمین که بدون رفتن به پنل اسپات پلیر، لایسنس رو مستقیماً غیرفعال یا فعال کنه (صفر/بازیابی device count).

44. **Import/Export تنظیمات**: دکمه «دانلود تنظیمات» که تنظیمات افزونه رو به JSON export کنه و دکمه «بارگذاری تنظیمات» برای import. خیلی مفیده برای migration بین سرورها یا backup قبل از تغییر کلید API.

    - نشان‌دهنده تاریخ خرید
    - badge وضعیت لایسنس (فعال / منقضی)
    - دکمه «شروع یادگیری» که مستقیم پلیر وب رو باز می‌کنه (اگه web فعال باشه)

46. **جستجو در دوره‌های کاربر**: یه فیلد جستجو روی شورت‌کد که کاربر بتونه بین دوره‌هایش سرچ کنه. برای کاربرانی که ۱۰+ دوره دارند خیلی مفیده.

47. **فیلتر وضعیت دوره**: در صفحه دوره‌های کاربر، tab هایی مثل «همه / فعال / منقضی» اضافه کن. نیاز به ذخیره تاریخ انقضا در local metadata داره.

---

### ⏱️ محدودیت زمانی و تاریخ انقضا

48. **تنظیم خودکار تاریخ انقضا بر اساس محصول**: در تب اسپات پلیر صفحه محصول، یه فیلد «مدت دسترسی (روز)» اضافه کن. هنگام ساخت لایسنس، تاریخ انقضا = تاریخ خرید + این تعداد روز محاسبه و در API ارسال بشه. این یه امکان کلیدی برای دوره‌های اشتراکی یا دوره‌های با دسترسی محدوده.

49. **نمایش تایمر معکوس به کاربر**: در صفحه دوره‌های کاربر، برای دوره‌هایی که تاریخ انقضا دارند یه نوار وضعیت نشون بده (مثلاً «۱۵ روز مانده») که با رنگ متفاوت urgent بشه اگه کمتر از ۷ روز مونده باشه.

---

### 📋 بهبود فرم‌های CSV

50. **پیش‌نمایش CSV قبل از اجرا (Dry Run)**: قبل از اجرای عملیات bulk، یه مرحله «پیش‌نمایش» اضافه کن که نشون بده چند ردیف پردازش خواهد شد، کدام‌ها کاربر موجود هستند و کدام‌ها جدید. کاربر تأیید کنه بعد اجرا بشه.

51. **دانلود Template CSV**: یه لینک «دانلود فایل نمونه CSV» زیر هر فرم بذار که یه فایل نمونه با header های درست رو برای کاربر دانلود کنه. الان کاربر باید header ها رو حفظ باشه.

52. **گزارش کامل‌تر بعد از bulk**: بعد از اجرای عملیات bulk، یه لینک «دانلود گزارش CSV» بده که نتیجه هر ردیف (موفق/خطا + شناسه لایسنس) رو داشته باشه. الان فقط متن نمایش داده می‌شه و اگه ۱۰۰ تا ردیف باشه قابل ذخیره نیست.

---

### 🔄 WooCommerce Subscriptions (عمیق‌تر از ایده ۲۳)

53. **تمدید خودکار تاریخ انقضا لایسنس هنگام renewal**: وقتی اشتراک تجدید می‌شه، نه‌تنها لایسنس باید فعال بمونه، بلکه تاریخ انقضا باید از طریق API تمدید بشه. این ترکیب ایده ۲۳ و ۴۸ است و برای آموزشگاه‌های اشتراکی حیاتیه.

---

### 🏷️ شورت‌کد پارامتری

54. **پارامترهای شورت‌کد**: شورت‌کد فعلی بدون پارامتره. امکانات زیر اضافه بشند:
    - `[spotplayer_courses product="123"]` — فقط دوره‌های مرتبط با یه محصول خاص
    - `[spotplayer_courses limit="6"]` — محدود کردن تعداد نمایش
    - `[spotplayer_courses layout="grid|list"]` — تغییر نوع نمایش
    - `[spotplayer_courses show_expired="false"]` — مخفی کردن منقضی‌ها

---

### 🏪 صفحه محصول و فروشگاه

55. **نمایش badge «داری» روی محصول**: اگه کاربر لاگین باشه و این محصول رو قبلاً خریده، یه badge «✓ قبلاً خریداری شده» روی کارت محصول نمایش بده. این برای فروشگاه‌هایی که کاربران ممکنه دوباره همون دوره رو بخرند خیلی مفیده.

56. **لینک «ادامه یادگیری» در صفحه محصول**: اگه کاربر لاگین باشه و لایسنس داشته باشه، دکمه «خرید» رو با «ادامه یادگیری» جایگزین کن که مستقیم به صفحه پلیر می‌ره.

---

### 🔒 کنترل دسترسی پیشرفته

57. **محدودیت IP برای پلیر وب**: گزینه‌ای در تنظیمات که بتونی IP range های مجاز برای استفاده از پلیر وب رو مشخص کنی. مثلاً فقط کاربران ایران. این با `$_SERVER['REMOTE_ADDR']` در `spot_shop_success` قابل پیاده‌ساییه.

58. **قفل نسخه وب به دستگاه**: الان پلیر وب با cookie کار می‌کنه. گزینه‌ای اضافه کن که پلیر وب به User-Agent مرورگر هم گره بخوره تا اشتراک‌گذاری cookie مشکل‌ساز نشه.

---

### 🛠️ ریز بهبودهای باکس ادمین

59. **QR Code کلید لایسنس**: در باکس سفارش ادمین، یه دکمه «نمایش QR» اضافه کن که کلید لایسنس رو به QR تبدیل کنه. کاربر می‌تونه با اسکن QR، کلید رو سریع‌تر وارد پلیر کنه.

60. **تاریخچه تغییرات لایسنس در order notes**: هر بار که لایسنس از طریق افزونه تغییر می‌کنه (ساخت، غیرفعال‌سازی، تمدید)، یه یادداشت با timestamp در سفارش ثبت بشه. الان ساخت ثبت می‌شه ولی غیرفعال‌سازی و تمدید ثبت نمی‌شند.

61. **نمایش نام دوره به جای شناسه**: در باکس سفارش ادمین، به جای شناسه دوره مثل `65a3f2...`، اسم دوره رو (با یه API call یا cache) نمایش بده.

---

### ⚡ بهبودهای فنی کوچک

62. **اضافه کردن `noindex` به صفحه pending**: وقتی کاربر صفحه تأیید سفارش رو با پیام «در حال آماده‌سازی» می‌بینه، این صفحه نباید توسط ربات‌های موتور جستجو ایندکس بشه. یه meta tag `noindex` به حالت pending اضافه کن.

63. **محافظت از `/spdeb` با IP whitelist**: endpoint دیباگ `/spdeb` الان فقط با بررسی `administrator` role محافظت می‌شه. محدود کردنش به IP ادمین سرور امنیت رو بالاتر می‌بره.

64. **پشتیبانی از محصولات Grouped در ووکامرس**: الان فقط Simple و Variable پشتیبانی می‌شند. اگه محصولات Grouped استفاده بشه (چند دوره در یه package) باید هر آیتم داخل group جداگانه بررسی بشه.

65. **Auto-detection نسخه WooCommerce HPOS**: افزونه باید در activation چک کنه که آیا HPOS فعاله یا نه و در صورت نیاز تابع مناسب رو صدا بزنه، به جای اینکه با `get_edit_post_link` شکست بخوره.

66. **کشینگ نتیجه `wc_get_products` در صفحه تنظیمات**: در صفحه تنظیمات، فرم ایجاد سفارش دسته‌ای `wc_get_products(['limit' => -1])` اجرا می‌کنه که روی سایت‌های با محصولات زیاد کند می‌شه. این رو باید با transient کش کرد.
